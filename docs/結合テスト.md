# 結合テスト実行手順

[← 要件定義書に戻る](requirements.md)

Controller→Service→Mapper→DB（本物のPostgreSQL 17）を通しで検証するAPI結合テストの実行手順。
CIは使用しないため、**PR作成前に手元で実行する**運用とする（CLAUDE.md 作業フロー4章）。

---

## 1. 前提

- **Docker Desktop が起動していること**（Testcontainers が PostgreSQL 17 のコンテナを自動起動するため）
- 開発用DB（`docker compose up -d` の `homelog-db`）とは独立したコンテナを使うため、開発用データは汚れない。事前の `docker compose up` も不要

## 2. 実行コマンド

```powershell
# backend/ ディレクトリで
.\gradlew.bat integrationTest
```

- 単体テスト（`.\gradlew.bat test`）はJUnitタグ`integration`を除外しており、従来通りDocker不要・高速なまま
- PR作成前は両方を実行する：

```powershell
.\gradlew.bat test integrationTest checkstyleMain checkstyleTest
```

## 3. テスト構成

`backend/src/test/java/com/homelog/integration/` に配置。全クラスが `IntegrationTestBase` を継承する。

| クラス | 検証内容 |
| --- | --- |
| `IntegrationTestBase` | 基盤。Testcontainers（`postgres:17`・staticシングルトンで共有）＋`TestRestTemplate`による実HTTP呼び出し。登録/ログイン/世帯作成のヘルパーを提供 |
| `AuthHouseholdFlowIT` | 登録→ログイン→世帯作成→別ユーザーが招待コード参加→両者が同一世帯。異常系：無効招待コード404・メール重複409・誤パスワード401・二重世帯作成400 |
| `ZaikoFlowIT` | 在庫登録→閾値下回りで買い物リスト自動追加（境界値：閾値ちょうどは対象外）→購入一括反映で在庫増加＆リスト除外→ダッシュボードサマリー反映。手動追加・削除・店舗その場登録 |
| `AuthorizationIT` | IDOR対策：他世帯の在庫・買い物リスト項目への操作が404。未認証は401 |

## 4. 設計上の注意

- テスト間のデータ独立は「テストごとに一意のメールアドレス・世帯を作成する」ことで担保している（`@Transactional`によるロールバックは実HTTP（RANDOM_PORT）では効かないため使わない）
- DBコンテナは全テストで1回だけ起動・共有される。スキーマとデフォルトカテゴリーはFlyway（`V1__init_mvp_schema.sql`）が投入する
- 新しい結合テストを追加する場合は `IntegrationTestBase` を継承すれば `@Tag("integration")` が自動で付く

## 5. トラブルシュート

| 症状 | 対処 |
| --- | --- |
| `Could not find a valid Docker environment` | Docker Desktop を起動する |
| 初回実行が遅い | `postgres:17` イメージのpullが走るため。2回目以降は数秒〜十数秒で完了する |
| ポート競合 | Testcontainersはランダムポートを使うため通常発生しない。発生時はDocker Desktopの再起動を試す |
