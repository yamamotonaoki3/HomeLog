## 1. プロジェクト概要

### 1.1 背景・目的

個人の財政管理を行いながら、1世帯内で協力して購入したものの精算・食材等の在庫管理・献立の計画を一括で行えるアプリ。詳細な背景・コンセプトは [構想.md](構想.md) を参照。

### 1.2 対象範囲

家計簿・在庫管理・献立表の3つの大きな機能領域を持つ。3機能のうち、在庫管理と買い物リストのみ密連携し、その他の機能間は独立して動作する（[common-notes.md](details/common-notes.md) 6章参照）。

---

## 2. プロジェクト体制・前提条件

### 2.1 前提条件

- 個人・家族・シェアハウス等、複数人同居での利用を想定
- 現段階では自分・家族での利用を想定するが、将来的には家族以外の第三者・別世帯にも一般公開し、複数の見知らぬ世帯グループが同一システム上で利用することを見据える
- モダンブラウザ（Chrome / Firefox / Edge 最新版）での利用を前提とする。将来的にはスマートフォン（モバイルアプリ）での動作にも対応したい

### 2.2 制約事項

- 認証はメールアドレス＋パスワードのみ（自前JWT認証、Amazon Cognito等のマネージド認証は使用しない）
- レシピ画像・手書き画像は AWS S3 に保存する
- インフラは AWS（ECS Fargate + RDS + S3 + CloudFront）を前提とした構成とする（詳細は[system-architecture.md](details/system-architecture.md)）

## 6. 非機能要件

### 6.1 性能要件

| 項目 | 要件 |
| --- | --- |
| レスポンスタイム | 通常操作で2秒以内（目安、詳細は今後検討） |
| 同時接続 | 初期フェーズは家族・世帯単位の少人数を想定 |

### 6.2 可用性要件

| 項目 | 要件 |
| --- | --- |
| 稼働目標 | 初期フェーズ（家族利用）では特定のSLAは設けない。全国展開時に再検討 |

### 6.3 セキュリティ要件

| 項目 | 要件 |
| --- | --- |
| 認証 | JWTアクセストークンによる認証（[技術スタック.md](技術スタック.md)参照） |
| パスワード | BCryptによるハッシュ化保存 |
| 通信 | HTTPS（本番環境） |
| 認可 | 個人の家計簿データは本人のみ操作可能。精算は相手承認制（[common-notes.md](details/common-notes.md) 2章） |
| データ分離 | 他世帯のデータに一切アクセスできないようにするアクセス制御（DB設計時に詳細を詰める） |

### 6.4 保守性要件

| 項目 | 要件 |
| --- | --- |
| コード品質 | Checkstyle（バックエンド）・oxlint（フロントエンド）によるチェック |
| マイグレーション | Flywayによる DB マイグレーション管理 |
| テスト方針 | テスト駆動開発（TDD）を採用し、機能ごとに単体テストが完結する構成とする（`CLAUDE.md`参照） |

### 6.5 移植性要件

| 項目 | 要件 |
| --- | --- |
| 対応環境 | Docker / docker-compose によるローカル環境構築 |
| 対応ブラウザ | Chrome / Firefox / Edge 最新版 |
| 対応デバイス | 初期はPC/スマートフォンのブラウザ（レスポンシブ対応）を前提とし、将来的にはネイティブ/クロスプラットフォームのモバイルアプリ化も検討する |

### 6.6 拡張性要件

| 項目 | 要件 |
| --- | --- |
| API設計 | REST API（Spring Boot）とフロントエンド（React）を分離した構成 |
| 将来のモバイル化 | WEBをまず構築し、将来的にモバイルアプリ化も検討する（構想.md参照） |
