# 全体共通事項

[← 要件定義書に戻る](../requirements.md)

---

機能横断で決まっている方針をここに集約する。個々の機能定義書で重複して記載しない。

## 1. 世帯グループモデル

- ユーザーは「世帯グループ」を作成・参加できる（[[../構想.md]] 2章）。
- 個人の財政（家計簿）は個人が管理し、世帯全体の合算家計簿は持たない。
- 世帯グループが管理するのは「協力購入・立て替えの精算関係」「在庫・買い物リストの共有」「献立表の共有」。
- シェアハウス等、家族に限らない複数人同居でも利用できることを想定。

## 2. 権限方針

| 機能 | 権限 |
| --- | --- |
| 個人の家計簿（支出・固定費・イベント） | 本人のみ編集可能。他の世帯メンバーは閲覧・編集不可 |
| 割り勘・精算 | 相手側（受領/請求の対象者）の承認があって初めて「精算済み」になる |
| 在庫管理・買い物リスト | 世帯メンバー全員が自由に編集可能（権限制限なし） |
| 献立表・レシピ | 世帯メンバー全員が自由に編集可能（権限制限なし） |

## 3. マスタデータ方針

- カテゴリーマスタは **機能ごとに独立** させ、共通化しない（家計簿用カテゴリーと在庫管理用カテゴリーは別物）。
- 在庫管理のカテゴリー・店舗マスタは世帯単位で共有し、使用中のものは削除不可（先に付け替えが必要）。

## 4. 通貨単位

- 支出データの通貨単位は日本円固定とする（多通貨対応は行わない）。

## 5. ID・命名規則

- 機能ID: `F-xx`（`docs/details/features/F0x_*.md` に対応）
- 画面ID: `S-xx`
- ダイアログ/モーダル: `D-xx` / `M-xx`

## 6. 機能間データ連携方針（まとめ）

| 連携元 | 連携先 | 連携有無 |
| --- | --- | --- |
| 在庫管理 | 買い物リスト | 密連携（同一データソース、自動追加・自動反映） |
| 買い物リスト | 家計簿 | 連携なし（支出は都度手動登録） |
| 献立表 | 在庫管理 | 連携なし（材料個数の自動消費は行わない） |
| 献立表 | 家計簿 | 連携なし |

## 7. 口座・カードモデル

- 「口座」（銀行口座・PayPay等の電子マネーを含む）を手動登録できる（[F11_kakeibo_account](features/F11_kakeibo_account.md)参照）。
- 「カード」（クレジットカード等）は口座に紐づく子エンティティとして扱う。カード単体では存在せず、必ずいずれかの口座に属する。
- 支出登録時に、どの口座・カードから出費したかを選択できる（`expenses.account_id`）。
- 外部API（銀行・カード会社・Money Forward ME・PayPay等）との自動連携は今回のスコープ外とし、手動登録のみを要件化する（今後の検討事項）。

## 8. 世帯合計支出の扱い

- 個人の支出データが「本人のみ閲覧・編集可能」という原則（2章）は維持する。
- ただし、世帯全体でどれだけ支出があったかの**合計金額**は世帯メンバー全員が確認できるようにする（[F12_kakeibo_household_summary](features/F12_kakeibo_household_summary.md)参照）。
- 個人的な支出（世帯の共同生活に関係しない私的な支出）は世帯合計から除外したいというニーズがあるため、支出・固定費それぞれに **世帯合計対象フラグ**（`include_in_household_total`）を持たせ、フラグがtrueのものだけを合計対象とする。
- 世帯合計に含める・含めないの判定はユーザーが支出登録時に都度指定する（自動判定ロジックは今回のスコープ外）。
- 世帯合計サマリーで開示するのは合計金額のみであり、「誰が何にいくら使ったか」という個別の内訳は非公開のまま。

## 9. 画面設計の基本方針（登録・編集は各機能画面、確認はトップに集約）

- 「今お金・生活まわりで何が起きているか」を都度画面遷移せずに把握できるよう、確認系の情報（今日の収支・今月の支出・未精算額・在庫不足・買い物リスト件数・イベント別支出等）は**トップ画面（S-04：生活ダッシュボード）に集約**する。
- 登録・編集操作は、各機能の専用画面・モーダル（S-06支出登録、S-18イベント登録等）で行う。
- 独立した「確認専用画面」は極力作らず、既存画面のサマリーカードやパネルとして表示する（例：世帯合計支出・イベント別支出サマリーはS-04に統合し、単独画面は持たない）。
- 密連携する機能（在庫管理と買い物リスト等）は、可能な限り同一画面内で同時に確認できるレイアウトにする。

## 10. APIエラーレスポンス方針

- **他人が所有するリソース（例：他人の支出ID、他世帯の在庫ID等）を指定した場合、403 Forbiddenではなく404 Not Foundを返す。**
- 理由：403を返すと「そのIDのリソースは存在するが権限がない」ことを外部に伝えてしまい、IDを総当たりすることで他人のデータの存在・件数・採番規則を推測されてしまう（IDOR/列挙攻撃）。404で統一することで「存在しない」のか「権限がない」のかを区別できなくし、こうした推測を防ぐ。
- 本人の認証自体が失敗している場合（トークンなし・無効・期限切れ）は401 Unauthorizedを返す。
- 入力値のバリデーションエラーは400 Bad Requestを返す。
- 詳細なAPIエンドポイント仕様は[api-design.md](api-design.md)を参照。

## 11. 入力初期値・デフォルト値

| 項目 | デフォルト値 | 備考 |
| --- | --- | --- |
| 支出・固定費の`include_in_household_total` | `false` | 個人支出として登録される。世帯合計に含めたい場合のみユーザーがONにする（8章参照） |
| 家計簿カテゴリー初期リスト | 食費／日用品／交際費／光熱費／住居費／通信費／医療費／趣味・娯楽／その他 | [F03_kakeibo_expense](features/F03_kakeibo_expense.md)参照 |
| パスワード強度 | 8文字以上、英字と数字を両方含む | [F01_auth](features/F01_auth.md)参照 |
| 割り勘の負担割合（2人） | 50% : 50% | [F04_kakeibo_warikan](features/F04_kakeibo_warikan.md)参照 |
| 割り勘の負担割合（3人以上） | 人数で均等割り（端数は代表者へ寄せる） | 例：3人なら33.34% / 33.33% / 33.33% |
